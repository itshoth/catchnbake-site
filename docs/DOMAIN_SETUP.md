# Domain + Deploy: catchnbake.com → Cloudflare Pages

**Date:** 2026-07-03

## 0. The recommendation (and why)

This repo has no hosting config today (no wrangler.toml, no CI, no CNAME — just a GitHub remote at `github.com/itshoth/catchnbake-site`). A static site of ~15 files needs nothing more than a free static host.

**Use Cloudflare Pages (free tier), connected to the GitHub repo, with nameservers moved from Namecheap to Cloudflare.** Reasons:

- **Free and unmetered** for static assets — no bandwidth bills if a species page goes viral.
- **Clean URLs are automatic**: `/privacy` serves `privacy.html`, `/fish/sheepshead` serves `fish/sheepshead.html`, and `.html` requests 308-redirect to the clean form. The iOS app's hardcoded `catchnbake.com/privacy` and `/terms` links work with zero configuration.
- **`404.html` and `_redirects` picked up automatically** (both already in the repo).
- **Apex-domain support requires Cloudflare DNS anyway**: Pages custom domains on a bare/apex domain (catchnbake.com without www) only work when the zone is on Cloudflare, via CNAME flattening. Keeping DNS at Namecheap would limit you to www-only. That's why the nameserver switch is the clean path, not a preference.
- Git-push deploys with per-branch preview URLs; free SSL.

The only thing Namecheap keeps doing is registration/renewal. That's fine — no transfer needed.

## 1. One-time setup (≈20 minutes, only you can do this)

### Step 1 — Cloudflare account + zone
1. Create a free account at dash.cloudflare.com.
2. **Add a domain** → `catchnbake.com` → Free plan.
3. Cloudflare scans existing DNS (there's nothing needed for the site — mail records, if any exist at Namecheap, will be imported; verify any custom email/MX records made it across before switching).
4. Cloudflare shows **two nameservers** (e.g. `ada.ns.cloudflare.com` / `bob.ns.cloudflare.com`). Keep this tab open.

### Step 2 — Nameserver switch at Namecheap
1. Namecheap → Domain List → **Manage** next to catchnbake.com.
2. **Nameservers** section → change from "Namecheap BasicDNS" to **Custom DNS**.
3. Paste the two Cloudflare nameservers → save (✓ checkmark).
4. Propagation: usually under an hour, worst case 24–48h. Cloudflare emails you when the zone is active.

### Step 3 — Create the Pages project
1. Cloudflare dashboard → **Workers & Pages → Create → Pages → Connect to Git**.
2. Authorize GitHub, select **itshoth/catchnbake-site**.
3. Build settings: production branch `main`, **build command: (leave empty)**, **build output directory: `/`** (the repo root is the site).
4. Deploy. You get a `<project>.pages.dev` URL — verify the site renders there.

### Step 4 — Attach the custom domains
1. In the Pages project → **Custom domains → Set up a custom domain** → `catchnbake.com`. Cloudflare creates the flattened CNAME record in your zone automatically — accept it.
2. Repeat for `www.catchnbake.com`.
3. The repo's `_redirects` file (`https://www.catchnbake.com/* → https://catchnbake.com/:splat 301`) then 301s all www traffic to the apex, so there is exactly one canonical host — matching every `<link rel="canonical">` on the site.

### Step 5 — SSL verification
1. Cloudflare issues a Universal SSL cert automatically once the domain is active (minutes).
2. SSL/TLS app → set encryption mode to **Full (strict)** (Pages origin supports it).
3. SSL/TLS → Edge Certificates → enable **Always Use HTTPS**.
4. Verify: `https://catchnbake.com`, `https://www.catchnbake.com` (→ 301 → apex), `http://catchnbake.com` (→ https) all land on the padlocked apex.

### Step 6 — Post-launch (recommended)
- **Google Search Console**: add a *Domain* property for catchnbake.com, verify via the DNS TXT record (Cloudflare DNS → Add record → TXT). Submit `https://catchnbake.com/sitemap.xml`.
- App Store Connect: support URL `https://catchnbake.com/support`, marketing URL `https://catchnbake.com`, privacy policy URL `https://catchnbake.com/privacy`.

## 2. Deploy flow (every change after setup)

```
git push origin main        # that's the whole deploy
```
Pages builds (no build step — it just uploads) and goes live globally in ~30 seconds. Any non-main branch push gets its own preview URL (`<hash>.<project>.pages.dev`) — useful for reviewing copy changes.

Rollback: Pages dashboard → Deployments → ⋯ → **Rollback to this deployment**.

## 3. Route → file mapping on this host

| Live URL | File in repo | Notes |
|---|---|---|
| `/` | `index.html` | landing page |
| `/privacy` | `privacy.html` | hardcoded in the iOS app — never rename |
| `/terms` | `terms.html` | hardcoded in the iOS app — never rename |
| `/guidelines` | `guidelines.html` | linked from hosted terms + app store review |
| `/support` | `support.html` | App Store Connect support URL |
| `/fish/sheepshead` etc. | `fish/<slug>.html` | generated by `scripts/build-species.js` |
| `/sitemap.xml`, `/robots.txt`, `/favicon.svg` | same-named files | served as-is |
| any unknown route | `404.html` | automatic |
| `https://www.catchnbake.com/*` | `_redirects` | 301 → apex |

`/privacy.html` → 308 → `/privacy` is automatic; don't link the `.html` forms anywhere.

## 4. If you ever refuse the nameserver switch (not recommended)

Keeping DNS at Namecheap means Cloudflare Pages can only validate `www.catchnbake.com` (CNAME `www` → `<project>.pages.dev`); the apex would need Namecheap's ALIAS/URL-redirect hacks that break HTTPS on the apex or add latency. Since the iOS app links to the **apex** (`catchnbake.com/privacy`), the apex must be first-class. Do the nameserver switch.
